# GitHub手順書

## 1. 初期設定（初回のみ）

### Gitのユーザー設定
```bash
git config --global user.name "GitHubのユーザー名"
git config --global user.email "GitHubに登録したメールアドレス"
```
- コミット時に「誰が変更したか」を記録するために必要
- `--global` なので一度設定すれば全プロジェクトで有効

### プロジェクトの始め方

#### 方法A: clone で始める（おすすめ）
1. GitHubのブラウザでリポジトリを作成
2. リポジトリページの「Code」ボタンからURLをコピー
3. ターミナルで以下を実行：
```bash
git clone https://github.com/ユーザー名/リポジトリ名.git
```
- `git init` と `git remote add` を自動でやってくれるので一番楽
- クローンしたフォルダに移動して作業を開始する

#### 方法B: git init で始める
既にローカルにフォルダがある場合はこちら：
```bash
git init
git remote add origin https://github.com/ユーザー名/リポジトリ名.git
```
- `git init` → 今あるフォルダをGitリポジトリにする
- `git remote add` → GitHubのリポジトリと接続する

※ どちらの方法でも、まずGitHubブラウザでリポジトリを作成する必要がある

## 2. 基本用語

| 用語 | 意味 |
|------|------|
| リポジトリ | ファイルと変更履歴をまとめた保管庫 |
| ステージ（add） | コミットに含めるファイルを選ぶ |
| コミット（commit） | 変更をローカルに記録する（セーブポイント） |
| プッシュ（push） | ローカルの記録をGitHubにアップロードする |
| プル（pull） | GitHubの最新を自分のPCにダウンロードする |
| クローン（clone） | GitHubのリポジトリをまるごとPCにコピーする |
| ブランチ（branch） | 作業の分岐。mainを壊さずに作業するための仕組み |
| プルリクエスト（PR） | ブランチの変更をmainに入れてほしいというレビュー依頼 |
| マージ（merge） | ブランチの変更をmainに取り込むこと |
| コンフリクト（競合） | 同じファイルの同じ箇所を複数人が変更した場合に発生。手動で解決が必要 |

---

## 3. 基本の流れ（個人開発）

```
ステージ(add) → コミット(commit) → プッシュ(push)
選ぶ            記録する            GitHubに送る
```

### ターミナルの場合
```bash
git add ファイル名      # 全ファイルの場合: git add .
git commit -m "変更内容のメモ"
git push
```

### CursorのGUIの場合
1. 左サイドバーの **Source Control（分岐マーク）** を開く
2. ファイル横の **「＋」ボタン** → ステージ（add）
3. メッセージ欄に変更内容を入力 → **コミットボタン（✓）**
4. **「変更の同期」または「↑」ボタン** → プッシュ

※ コミット時にメッセージを空にするとエディタが開いて止まることがあるので注意

## 4. チーム開発の流れ（ブランチを使う）

### 基本ルール
- **mainブランチには直接コミットしない**
- 必ずブランチを作って作業する

### 手順

#### 4-1. ブランチを作る

**ターミナルの場合：**
```bash
git checkout -b feature/新機能名
```

**Cursorの場合：**
1. 左下の「main」をクリック
2. 「新しいブランチの作成...」を選ぶ
3. ブランチ名を入力（例：`feature/test3`）
4. Enterを押す

#### 4-2. 作業してpushする
```bash
git add .
git commit -m "○○を追加"
git push -u origin feature/新機能名
```
※ Cursorでは「Branchの発行」ボタンで初回pushできる

#### 4-3. プルリクエスト（PR）を作る
1. GitHubのブラウザでリポジトリを開く
2. 「Compare & pull request」をクリック
3. タイトルと説明を入力
4. 「Create pull request」をクリック

#### 4-4. レビュー＆マージ
1. チームメンバーがコードを確認・承認する
2. 「Merge pull request」を押してmainに合流させる

#### 4-5. mainを最新にする
```bash
git checkout main
git pull
```

---

## 5. マージについて

- マージはブランチの変更差分をmainに取り込む操作
- mainに変更がなければ、ブランチの内容がそのまま反映される
- 同じファイルの同じ箇所をmainとブランチの両方で変更した場合、コンフリクトが発生する
- 「No conflicts with base branch」と表示されていれば安全にマージできる

---

## 6. ブランチの命名規則

```
feature/○○    ← 新機能
fix/○○        ← バグ修正
hotfix/○○     ← 本番の緊急修正
```

---

## 7. 全体の流れまとめ

### 個人開発
```
ファイル編集 → ステージ → コミット → プッシュ
```

### チーム開発
```
ブランチ作成 → 作業 → ステージ → コミット → プッシュ → PR作成 → レビュー → マージ → mainをpull
```
